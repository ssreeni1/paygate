## Confidence Tests

### The 2am Friday Test
"Can I deploy and trust that no payment gets verified twice and no valid payment gets rejected?"
- Test 4 (replay rejection under concurrency)
- Test 1 (cross-language hash parity)
- Load test: 1000 concurrent payments, verify exactly 1000 unique entries in SQLite

### The Hostile QA Test
- Send tx with TWO matching Transfer events → must reject (ambiguous)
- Send tx where `to` is token contract address, not provider → must reject
- Send tx from address A, claim with X-Payment-Payer: B → must reject
- Send valid payment, replay 100 times concurrently → exactly 1 acceptance
- Send payment with memo from a different quote → must reject

### The Chaos Test
- Kill gateway mid-verification → consumer retries → tx still valid
- Fill disk → 503 (not 500, not crash)
- Point RPC at wrong chain → all verifications fail gracefully

## Flakiness Risks

| Test | Risk Source | Mitigation |
|------|------------|------------|
| E2E testnet tests | Tempo RPC availability | Use recorded responses for CI; real RPC for `paygate test` only |
| Timing-dependent tests | Clock drift | Use monotonic clocks, allow 1s tolerance |
| SQLite concurrency | File locking | Use in-memory SQLite for unit tests |
| Webhook delivery | Network | Mock HTTP server in tests |

## Pre-Implementation Verification Task

**CRITICAL:** Before writing any code, verify that the following Tempo SDK APIs exist and match the spec:
- `viem/tempo`: `Account.fromSecp256k1`, `tempoActions()`, `withFeePayer`
- `viem/chains`: `tempoTestnet` / `tempoMainnet` chain objects
- TIP-20 `Transfer(address,address,uint256)` event ABI
- TIP-20 `TransferWithMemo` event ABI and memo field encoding
- `tempo_fundAddress` RPC method for testnet faucet

If any of these don't exist or differ, the spec must be updated before implementation.
